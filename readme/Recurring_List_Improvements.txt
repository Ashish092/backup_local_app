KEY CHANGES
================================================================================

1. DATA FETCHING LOGIC
   -------------------
   Old approach:
   - Fetched booking_status_on_app.assigned_to (master level)
   - Showed master booking's worker (incorrect for upcoming)
   
   New approach:
   - Fetches upcoming instance from recurring_booking_status table
   - Filters instances where instance_date >= today
   - Gets first upcoming instance per booking
   - Shows that instance's status, worker, and date
   
   Query:
   ```sql
   SELECT master_booking_id, instance_date, job_status, assigned_to
   FROM recurring_booking_status
   WHERE master_booking_id IN (...)
     AND instance_date >= '2026-03-11'
   ORDER BY instance_date ASC
   ```
   
   Then groups by master_booking_id and takes first match.

2. INTERFACE CHANGES
   ------------------
   Updated RecurringBooking interface:
   
   Added:
     - upcoming_status: string (from upcoming instance)
     - upcoming_worker: string | null (from upcoming instance)
     - upcoming_date: string | null (from upcoming instance)
     - customer_phone: string | null
   
   Deprecated (kept for compatibility):
     - job_status (was master level)
     - assigned_to (was master level)
     - worker_name
   
   Removed:
     - instance_count: number

3. CARD LAYOUT
   ------------
   New vertical structure:
   
   ┌─────────────────────────┐
   │ #CH_0080    [Service]   │  ← Top: Booking # + Service

   --------------------
   Now searches:

5. FILTER LOGIC
   -------------

6. STATS ROW

   Before: Total | Assigned | Open | Instances

   Calculation now uses upcoming_status instead of master job_status
